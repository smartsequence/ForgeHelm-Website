## 📋 修改說明

### 主要調整項目

#### 1. 標題改寫

```
原：讓老系統說人話，讓風險一目了然
改：讓老系統的風險看得見，讓交接不再像猜謎

原因：
- 「說人話」太口語，不夠專業
- 「一目了然」是成語，有點正式
- 「像猜謎」更貼近真實痛點
```

#### 2. 移除過度包裝

```
原：智慧化風險報告產出工具
改：專門用來分析系統風險的工具

原因：
- 「智慧化」是 AI 生成常用詞
- 「產出」太書面語
- 更直白說明用途
```

#### 3. 改用自然語氣

```
原：只要填寫一份導入問卷，Doc Engine 即可自動產出
改：只要填一份問卷，Doc Engine 就會自動產生

原因：
- 移除「導入」這種商業術語
- 「即可」改「就會」更口語
- 「產出」改「產生」更自然
```

#### 4. 調整表情符號使用

```
原：🔍 已推出首版 SaaS 工具
改：• 已推出第一版 SaaS 工具

原因：
- 表情符號太多會顯得不專業
- 改用項目符號（•）更簡潔
- 「首版」改「第一版」更口語
```

#### 5. 簡化句子結構

```
原：可作為一次性交付報告或後續產品採購的前導入口
改：可以當作一次性的交付報告，或是後續採購的參考

原因：
- 「前導入口」太正式
- 分成兩段更好讀
- 「當作」「或是」更口語
```

#### 6. 價格表述更直白

```
原：我們提供簡單透明的價格，方便您快速啟用報告流程
改：簡單透明的價格，不複雜

原因：
- 移除多餘的客套話
- 「啟用報告流程」太書面
- 直接說「不複雜」更有信任感
```

#### 7. CTA 按鈕文字更親切

```
原：需要更多資訊？
改：有問題？問我們

原因：
- 「資訊」太正式
- 「問我們」更直接友善
```

## ✅ 修改原則總結

### 避免使用的詞彙（AI 味重）

- ❌ 智慧化、可視化、產出
- ❌ 即可、導入、前導
- ❌ 賦能、推動、落地
- ❌ 一目了然、一應俱全
- ❌ 過多的表情符號

### 建議使用的詞彙（自然流暢）

- ✅ 分析、產生、看得見
- ✅ 就會、可以、或是
- ✅ 幫您、讓您、給您
- ✅ 摸清楚、有問題、不複雜
- ✅ 適當的項目符號（•）

### 語氣風格

```
原風格：正式、包裝、推銷感重
新風格：直接、誠實、像在對話

例如：
原：我們專注於系統交付風險的分析與可視化
改：我們專注在系統交付時的風險分析

原：打造 Doc Engine 風險指紋報告
改：透過 Doc Engine 產生風險分析報告
```
